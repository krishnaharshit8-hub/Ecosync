0.1 — What EcoSync Is
EcoSync is an AI-driven orchestrator for decentralized microgrids that enables buildings to autonomously trade
solar power with each other using blockchain-secured smart contracts. The system uses NVIDIA Isaac Sim as a
Digital Twin to simulate 5 houses, LSTM neural networks to predict energy production and consumption with
95% accuracy, LangGraph multi-agent systems for autonomous negotiation between buildings, Solidity smart
contracts on Polygon Mumbai testnet for trustless trade execution, and a React.js live dashboard for monitoring.
One-line purpose: Every building becomes a smart node that predicts its own energy needs and intelligently
trades surplus solar power with neighbors — like an Airbnb for electricity.

1.2 — The Vision
A microgrid is a localized group of electricity sources and consumers that normally operates connected to the
traditional wide-area grid, but can disconnect and function autonomously (called 'island mode') when needed. In
EcoSync, the microgrid is a neighborhood of 5 buildings each with solar panels and battery storage.
Decentralized means energy decisions are made by the buildings themselves — not a central utility company.
Each building has its own AI agent that monitors its energy state and negotiates trades autonomously. No
central authority is needed.
Why AI + Blockchain? AI (LSTM) predicts when a building will have surplus or deficit energy before it happens,
enabling proactive trading. Blockchain executes the financial settlement of trades automatically, without a bank
or utility billing system. Neither party can cheat — the smart contract only releases payment if the trade
agreement is met.
The Airbnb Mental Model: Just as Airbnb lets homeowners rent spare rooms to travelers without a hotel
company in the middle, EcoSync lets buildings sell spare solar watts to neighbors without a utility company in
the middle. The platform is the AI+blockchain stack; the 'hosts' are solar-equipped buildings; the 'guests' are
energy-needy buildings.
1.3 — Core Problems Being Solved
Grid Instability Crisis

Modern cities are growing faster than grid infrastructure. Traditional centralized grids were designed for
one-directional power flow (power plant → home). With solar panels, homes now produce power too, creating
bidirectional flows the old grid can't handle. Result: brownouts, blackouts, and expensive emergency upgrades.
EcoSync solves this by managing energy locally so less stress hits the main grid.
The 30% Renewable Waste
When solar panels produce more power than the local grid can absorb, the excess is simply curtailed (switched
off). This is called curtailment. Globally, 30% of potential renewable energy is wasted this way. EcoSync creates
a local marketplace so surplus solar is traded to neighbors instead of wasted.
The Sun/Wind Intermittency Gap
Solar peaks at noon; electricity demand peaks at 6-9pm. This gap means buildings can't rely on solar alone.
EcoSync's LSTM model predicts this gap hours in advance, allowing agents to proactively charge batteries at
noon and sell/draw from neighbors at 6pm.
Cyber Vulnerability
Smart meters and IoT grid devices are prime cyberattack targets. A centralized grid has a single point of failure.
EcoSync's decentralized approach means each building's agent operates independently — compromising one
node doesn't take down the whole grid. Blockchain's immutable ledger also prevents trade manipulation or data
falsification.
1.4 — Solution Summary (4 Pillars)
Predictive AI (LSTM)
Long Short-Term Memory networks analyze the past 3 hours of energy data (12 readings × 3 features) and
predict the next hour's production and consumption for each house. Target accuracy: 95%. This gives agents
enough warning to negotiate trades before energy is actually needed.
Multi-Agent Orchestration (LangGraph)
Each building has one autonomous LangGraph agent. Agents assess their own energy state, decide to BUY,
SELL, or IDLE, then enter a negotiation phase where they match with compatible partners. The orchestrator
runs all agents simultaneously and executes matched trades.
P2P Blockchain Trading (Solidity + Polygon)
Once agents agree on a trade, the EnergyTrade.sol smart contract on Polygon Mumbai executes the financial
settlement. The executeTrade(seller, buyer, amount, price) function only runs if both parties' conditions are met.
All trades are permanently recorded on-chain with a transaction hash for full auditability.

SECTION 2
System Architecture
2.1 — Full Architecture Overview
EcoSync is a 5-layer cyber-physical stack. Each layer handles a distinct concern and communicates with
adjacent layers via well-defined protocols. Data originates in the IoT/Simulation layer and flows upward through
AI, negotiation, and blockchain before being visualized on the dashboard. The stack is designed so each layer
can fail independently without bringing down the others.
2.2 — Layer-by-Layer Breakdown
Layer 1: DATA LAYER — IoT & Digital Twin
Property
Details
Purpose
Provides real-time energy production and consumption data for all 5 houses.
Input

JSON-formatted sensor data published to MQTT topics every 15 simulated minutes
Technologies
NVIDIA Isaac Sim, Python solar_generator.py, Mosquitto MQTT broker, Paho-MQTT client
Communication
Publishes to ecosync/houses/H1 through H5 on Mosquitto port 1883
Failure Mode
System reverts to historical average values from PostgreSQL for predictions
Layer 2: INTELLIGENCE LAYER — AI Prediction
Property
Details
Purpose
Forecasts each building's energy state 1 hour into the future.
Input
Last 3 hours of (production_kw, consumption_kw, battery_soc) = shape (12, 3)
Output
Predicted [production_kw, consumption_kw] for next hour per house
Technologies
TensorFlow 2.13, Keras LSTM layers, scikit-learn MinMaxScaler, FastAPI inference endpoint
Communication
Backend calls lstm_model.predict() internally; results stored in PostgreSQL predictions table
Failure Mode
Falls back to persistence model (next hour = last hour). Log the fallback in agents_log table.

class ConnectionManager:
def __init__(self):
self.active: list[WebSocket] = []
async def connect(self, ws: WebSocket):
await ws.accept()
self.active.append(ws)
def disconnect(self, ws: WebSocket):
self.active.remove(ws)
async def broadcast(self, data: dict):
msg = json.dumps(data)
dead = []
for ws in self.active:

except Exception:
dead.append(ws)
for ws in dead:
self.active.remove(ws)

6.1 — LSTM Architecture
The LSTM model solves a sequence-to-value regression problem: given the last 12 readings (3 hours × 4
readings/hour) of 3 features (production_kw, consumption_kw, battery_soc), predict [production_kw,
consumption_kw] for the next hour. This gives each agent enough time to negotiate trades before the predicted
state occurs.

7.1 — Blockchain Concepts for Agents (Zero Prior Knowledge)
Term
Plain English Explanation
Smart Contract
A program stored on a blockchain. Once deployed, its code cannot be changed. It runs
automatically when called. Think of it as a vending machine: put in the right input, get the
guaranteed output, no human needed.
Polygon / Mumbai
Polygon is a fast, cheap Ethereum-compatible blockchain. Mumbai is its public test network
(testnet). On Mumbai, transactions cost fake MATIC tokens (free from a faucet), so we can test
without spending real money.
Solidity
The programming language for writing smart contracts. Compiled to bytecode that runs on the
Ethereum Virtual Machine (EVM). Files end in .sol
Wallet / Private Key
A wallet is a pair of keys: a public address (like a bank account number) and a private key (like
a PIN). The private key signs transactions. NEVER share it. Store only in .env.
Transaction Hash
(tx_hash)
A unique 66-character fingerprint of a transaction on the blockchain. Format: 0x followed by 64
hex chars. You can look it up on mumbai.polygonscan.com to verify it happened.
ABI
Application Binary Interface — a JSON description of a smart contract's functions. Used by
Web3.py to know how to call the contract from Python.

import numpy as np
import math
from datetime import datetime, timedelta
HOUSE_CONFIG = {
'H1': {'panel_kw': 5.0, 'efficiency': 0.20, 'base_consumption': 2.5},
'H2': {'panel_kw': 4.5, 'efficiency': 0.18, 'base_consumption': 2.2},
'H3': {'panel_kw': 6.0, 'efficiency': 0.22, 'base_consumption': 4.0},
'H4': {'panel_kw': 8.0, 'efficiency': 0.21, 'base_consumption': 6.0}, # MedCenter
'H5': {'panel_kw': 5.5, 'efficiency': 0.19, 'base_consumption': 2.8},
}
def solar_irradiance(hour: float, cloud_cover: float) -> float:
'''Calculate solar irradiance (0-1) based on time of day and clouds.
Peak at hour 12 (noon). Zero before 6am and after 7pm.
'''
if hour < 6 or hour > 19:
return 0.0
# Sine curve peaking at noon
angle = math.pi * (hour - 6) / (19 - 6)
base = math.sin(angle)
cloud_factor = 1.0 - (cloud_cover * 0.8) # Clouds reduce by up to 80%
noise = np.random.normal(0, 0.05) # Small random variation
return max(0, min(1.0, base * cloud_factor + noise))
def simulate_house(house_id: str, hour: float, cloud_cover: float,
battery_soc: float) -> dict:
'''Simulate one 15-minute tick for a house.'''
cfg = HOUSE_CONFIG[house_id]
irradiance = solar_irradiance(hour, cloud_cover)
production = round(cfg['panel_kw'] * cfg['efficiency'] * irradiance * 5, 3)
# Consumption: base + random peak (evening hours have higher consumption)
evening_factor = 1.0 + 0.5 * max(0, math.sin(math.pi * (hour - 14) / 8))
consumption = round(
cfg['base_consumption'] * evening_factor * np.random.uniform(0.85, 1.15), 3
)
# Battery: simple charge/discharge model
net = production - consumption
battery_change = net * 0.95 # 95% charge efficiency
new_battery = max(0, min(100, battery_soc + battery_change * 10))
return {
'house_id': house_id,
'timestamp': datetime.utcnow().isoformat() + 'Z',
'solar_production_kw': production,
'consumption_kw': consumption,
